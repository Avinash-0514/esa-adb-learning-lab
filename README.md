## Overview

`esa-adb-learning-lab` documents my learning journey with the **European Space Agency Anomaly Detection Benchmark (ESA-ADB)**. The repository focuses on learning how raw satellite telemetry is organised, how anomaly labels and telecommands relate to telemetry channels, and how large time-series data can be transformed into reliable, model-ready information.

This is intentionally a **learning and experimentation repository**. It records the questions, code, mistakes, corrections, observations, and engineering decisions made while progressing from raw data exploration towards an anomaly-detection pipeline.

The final production-oriented anomaly-detection application will be maintained separately.

## Learning objectives

The main objectives are to:

- understand the structure of multivariate satellite telemetry;
- work confidently with Python, NumPy, pandas, Matplotlib, and Jupyter;
- analyse telemetry channels at dataset, channel, period, and event levels;
- connect anomaly labels, telemetry channels, telecommands, and subsystem metadata;
- design repeatable preprocessing steps without loading the entire dataset unnecessarily;
- identify missing values, duplicated records, irregular sampling, long gaps, and time-zone issues;
- engineer useful statistical and time-based features;
- prevent data leakage when preparing training, validation, and test data;
- gradually build a model-ready data pipeline for time-series anomaly detection.

## Dataset

This project uses the **ESA Anomaly Detection Benchmark**, which contains real spacecraft telemetry, anomaly annotations, and supporting metadata prepared for time-series anomaly-detection research.

Primary sources:

- [ESA-ADB GitHub repository](https://github.com/kplabs-pl/ESA-ADB)
- [ESA Anomalies Dataset on Zenodo](https://doi.org/10.5281/zenodo.12528696)

The current exploration begins with **Mission 1** and considers four important data groups:

| Data group | Purpose |
|---|---|
| Telemetry channels | Time-series measurements generated by spacecraft systems |
| Anomaly labels | Start time, end time, event ID, and affected channel information |
| Telecommands | Commands sent to the spacecraft that may explain changes in telemetry |
| Metadata | Channel, anomaly-category, and subsystem information |

The original ESA-ADB files are **not stored in this repository**. They should be downloaded from the official source and kept locally according to the dataset licence and documentation.

## Exploration strategy

The dataset is studied through four complementary views.

### View 1 — Dataset overview

Purpose: build a map of the complete dataset before analysing individual signals.

- inspect available missions and files;
- count channels, labels, telecommands, and subsystem records;
- examine schemas, column names, data types, and date ranges;
- understand relationships among IDs, channels, categories, and subsystems;
- estimate file sizes and decide which files should be loaded on demand.

### View 2 — Channel-wise analysis

Purpose: understand the behaviour and data quality of one telemetry channel at a time.

- sample count and time coverage;
- minimum, maximum, mean, median, and standard deviation;
- unique and frequently repeated values;
- missing values and duplicate timestamps;
- sampling intervals and long gaps;
- constant or near-constant behaviour;
- trend, variance, spikes, and possible state changes.

### View 3 — Time-period analysis

Purpose: examine what happened across the spacecraft during a selected period.

- select a day, week, or month;
- find anomaly events overlapping the period;
- identify affected telemetry channels;
- collect telecommands issued within the same window;
- compare activity across channels and subsystems;
- investigate whether the period is quiet, command-heavy, or anomaly-heavy.

### View 4 — Event-wise analysis

Purpose: study a labelled anomaly in its operational context.

- select one or more anomaly IDs;
- create before, during, and after windows;
- load only the affected channels;
- compare phase-level statistics;
- align telecommands with telemetry timestamps;
- inspect onset, duration, propagation, and recovery;
- record observations without assuming that correlation proves causation.

## Preprocessing roadmap

The preprocessing pipeline will be developed incrementally and verified at every stage.

| Stage | Main questions | Planned output |
|---|---|---|
| 1. Schema validation | Are the expected columns and data types present? | Validation report |
| 2. Time normalisation | Are timestamps parsed consistently and aligned to UTC? | Clean time index |
| 3. Quality checks | Are values missing, duplicated, invalid, or out of order? | Data-quality summary |
| 4. Sampling analysis | Are channel intervals regular or irregular? | Gap and frequency statistics |
| 5. Window construction | What context is required before and after an event? | Event-centred slices |
| 6. Resampling and alignment | How should channels with different frequencies be combined? | Aligned feature table |
| 7. Missing-data treatment | Does a missing value mean data loss, inactivity, or a different sampling rate? | Documented strategy |
| 8. Scaling | Should values be standardised globally, per channel, or with robust statistics? | Scaled features |
| 9. Feature engineering | Which changes, rolling statistics, or contextual signals are useful? | Model-ready features |
| 10. Dataset splitting | How can chronological separation prevent leakage? | Train/validation/test sets |

### Preprocessing principles

- preserve the original raw data;
- parse and sort timestamps before time-based filtering;
- use timezone-aware timestamps consistently;
- avoid treating every missing timestamp as a missing measurement;
- fit scaling and imputation parameters on training data only;
- split time-series data chronologically rather than randomly where appropriate;
- retain event IDs and channel names for traceability;
- document every transformation and its reason;
- validate intermediate outputs instead of relying only on the final table.

## Candidate features for future modelling

Feature engineering will be driven by observed signal behaviour rather than added blindly.

| Feature family | Examples | What it may reveal |
|---|---|---|
| Level | raw value, mean, median | operating state |
| Spread | standard deviation, range, IQR | instability or unusual variation |
| Change | first difference, percentage change, slope | drift or sudden transitions |
| Rolling | rolling mean, rolling standard deviation, rolling min/max | local context |
| Temporal | hour, day, elapsed time, time since previous sample | periodic or operational patterns |
| Gap | sampling interval, missing-run length | telemetry interruption |
| Command context | command count, time since command, command type | possible operational influence |
| Cross-channel | correlation, lagged relationship, subsystem aggregation | propagated or shared behaviour |

These features will be tested carefully because a feature that looks useful during an anomaly may accidentally contain future information or label leakage.

### Current observations

- Mission 1 contains long-running telemetry channels with very large row counts.
- Different channels may have different sampling behaviour and therefore should not be aligned without investigation.
- Some signals remain close to a repeated operating value for long periods.
- Telecommands are sparse events and require event-based handling rather than ordinary continuous-series assumptions.
- Before/during/after statistics can expose changes in signal level and variability, but plots and operational context are still needed for interpretation.
- A telecommand occurring near an anomaly is evidence for investigation, not proof that the command caused the anomaly.

## Running the notebooks

### 1. Clone the repository

```bash
git clone https://github.com/Avinash-0514/esa-adb-learning-lab.git
cd esa-adb-learning-lab
```

### 2. Create and activate a virtual environment

```bash
python3 -m venv .venv
source .venv/bin/activate
```

### 3. Install the core packages

```bash
python3 -m pip install jupyter pandas numpy matplotlib
```

### 4. Start Jupyter

```bash
jupyter notebook
```

Download the ESA-ADB data separately and update local dataset paths as required. Dataset archives, extracted telemetry files, virtual environments, and secrets should remain excluded from Git.
